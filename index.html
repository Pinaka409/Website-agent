3 · Agent input behaviour

User inputs sharpen the estimate further — but only as optional, single-tap refinements. Domain stays the only required input. Signup form stays mounted.

The refinement principle

Every additional input the agent collects is one extra friction point against signup. The accuracy gain has to be worth more than the conversion cost. Three rules:

  1. Domain alone always works. The agent delivers a complete pitch and signup form on domain alone. Refinements are upsell, never blockers.
  2. Single-tap inputs only. No free text, no multi-turn questions. Chips and toggles. If a user can't answer in one tap, the question doesn't ship.
  3. Live recalibration. Every chip tap updates the displayed range immediately. The user sees the number move; that's the dopamine that earns the second tap.

What we ask for, and why

Four optional dimensions, ranked by accuracy lift per tap. Ship them in order; stop when conversion data says the next one isn't earning its cost.

InputWhy it mattersFormat
Monthly ad spend Replaces the most fragile variable in the formula (traffic-derived spend) with a self-reported anchor. Single biggest accuracy lift. 5 chips: <$5K, $5–25K, $25–100K, $100–500K, $500K+
Employee count Confirms or corrects the firmographic signal. Drives SEATID directly and tunes ADID/KLOUDID size assumptions. 4 chips: 1–10, 10–50, 50–250, 250+
Vertical confirm BuiltWith infers vertical from stack but gets it wrong on edge cases (B2B SaaS on Shopify, agencies on WordPress). Confirmation costs nothing and prevents a class of bad estimates. Pre-selected chip from inference + 3–4 alternatives

4 · How refinement updates the agent

Backend flow when a chip is tapped. Same telemetry hook that logs estimate-vs-actual now also logs estimate-vs-self-reported, which gives us a second feedback loop.

1
Initial estimate runs on domain alone
BuiltWith + business profile + benchmark model produce the seed range. save_website_info caches it. Signup form mounts immediately.
2
User taps a chip
Frontend POSTs { session_id, dimension, value } to /refine_estimate. No model call needed for the recompute — it's a deterministic re-key against the benchmark model with the user's anchor as the override.
3
Backend recalibrates and returns new range
Order of precedence: user inputs override inferred values. New range and savings projection returned in same response. Sub-100ms target.
4
Frontend animates the number
Bolded number in the agent message animates to the new value (250ms ease). Recovery range below animates in parallel. Source receipt updates if the new estimate draws on different cohort sample sizes.
5
Telemetry logs the override
User-supplied value, original inferred value, and dimension stored. After signup + connect, we get the actual — three-way comparison feeds the calibration loop quarterly.

5 · Behaviour rules for input handling

When inputs disagree with inference, when to ask, when to stay silent.

1
User input always wins

If the user reports $25–100K ad spend and BuiltWith inferred small, the user is right. Override silently. Never argue with self-reported data.

2
Show the receipt change

When inputs change the cohort, the source line updates. "Based on 47 similar accounts" → "Based on 12 similar accounts at your reported scale." Smaller N is honest, not weak.

3
Never block on inputs

Signup form is always tappable. The refinement panel collapses when scrolled past. If the user signs up without tapping a chip, that's a successful conversion, not a missed step.

4
Big disagreements → soft surface

If user reports $500K+ when we inferred <$5K, the agent surfaces a one-liner: "That's significantly above what we usually see for your stack — connecting your account will give you the exact number." Acknowledges the gap without questioning the user.

5
No follow-up questions

Chips are the entire input surface. Agent never asks "can you tell me more about your business?" or "what's your revenue?" The flow remains one-shot. If we need more, we get it post-signup.

6
Carry refinements forward

Self-reported values join tech_profile and business_profile in the session row, then promote to the website record on signup. Onboarding agent picks up where pre-signup left off — no re-asking.

The conversion math

If 30% of visitors tap at least one chip and the chips lift estimate accuracy from ±60% to ±25%, the post-signup connect rate should improve materially because the pre-signup number now matches the post-connect actual closely enough to feel honest. The metric to watch: ratio of pre-signup estimate to post-connect actual. Target band: 0.7×–1.3×.

Spend range fades to $25K–$100K/month (250ms ease). Recovery range fades to $3K–$12K/month. Source line updates to "Based on 12 at your reported scale."
Behaviour rules
  • User input always wins. If the user reports $25K when we inferred $5K, the user is right. Override silently.
  • Live recalibration. No "click to update." Numbers animate immediately on tap (250ms ease).
  • Cohort-aware source line. When the chip shifts the matching cohort, N updates. Smaller N is honest, not weak.
  • Deterministic recompute. No model call needed — re-key against the benchmark with user input as override. Sub-100ms.
  • Signup form stays mounted. Chip taps never push the form below the fold or remount it.

4 · State 3 — big-disagreement soft surface

User input is wildly inconsistent with what we inferred from BuiltWith and firmographics. Agent acknowledges the gap without questioning the user.

3 Big-disagreement soft surface trigger: gap above threshold
What the user sees
User reports $500K+ spend on a stack we inferred as small ($5K–25K range).

Agent surfaces an amber-bordered note below the main pitch:

"That's significantly above what we usually see for your stack — connecting your account will give you the exact number."
Behaviour rules
  • Threshold for triggering. Soft surface fires when user-reported value is >3× the inferred top of range.
  • Acknowledge, don't question. Never "are you sure?" Always "connecting will give you the exact number."
  • Soft styling, not alarm. Amber accent border, not red. Information, not warning.
  • Drives toward connect. The disagreement becomes a connect-to-validate prompt, not friction.
  • Numbers still recalibrate. The user's input still overrides — the soft surface is additional, not blocking.
  • Auto-collapses if user revises down. If they tap a smaller chip after, the alert hides on its own.

5 · State 4 — multi-ID expansion

Below the main pitch, the agent surfaces the largest opportunity by ID category. Two CTAs — connect to drill into that ID, or pivot laterally to the next one.

4 Multi-ID expansion trigger: post-pitch render
What the user sees
"Top opportunity"

"Your largest opportunity is Ad ID at $3K–$8K/month. Connect Google Ads to break down which campaigns are typically driving the waste."

[Connect Google Ads →]   [Show me Seat ID instead]
Behaviour rules
  • Always lead with the largest dollar opportunity. Rank IDs by inferred value; surface the top one first.
  • Two CTAs — primary and lateral. Primary drives connect for that ID. Lateral pivots to the next biggest opportunity.
  • Lateral is a soft pivot, not dismissal. "Show me Seat ID instead" — they're still in the funnel, just looking at a different category.
  • Post-connect, expansion continues. After the user connects one platform, agent surfaces the next biggest opportunity. Progressive commitment.
  • Never push expansion before first connect. Pre-connect, the funnel goal is the first connect. Don't pre-empt.

6 · State 5 — connect-to-validate handoff

The close. Primary CTA drives signup + OAuth in one step. Secondary drops the user into email capture if they're not ready to connect.

5 Connect-to-validate handoff trigger: signup CTA tap
What the user sees
"See the exact recoverable amount"

[Connect & start audit →]

Email me the report instead
Behaviour rules
  • Connect-to-validate is primary. Replace the email-PDF download as the primary path. Email + PDF stays available as secondary.
  • Reframe the close. Not "see your report" — "see the exact recoverable amount." Specific outcome, not generic deliverable.
  • Pre-fill from session. Self-reported chips and the detected sizing both promote into the signup record. Onboarding agent picks up where pre-signup left off.
  • Connect button = OAuth flow. One tap to OAuth Google Ads (or Meta, or Shopify). No intermediate "create account" step.
  • Refusal handling. If user picks email-me-instead, capture the email and let them skip OAuth — fall back gracefully to estimate-first onboarding.

8 · State 7 — confidence label

Per marketing's signal-quality formula, every category gets a High / Medium / Low label based on which signals returned data. Low-confidence categories suppress the dollar range entirely.

7 Confidence label per ID trigger: signal coverage check
What the user sees
"Based on patterns across 47 similar audited accounts
HIGH confidence"

For LOW confidence: "Connect for exact — limited data on similar companies in your space."
Scoring formula (marketing's spec)
  • +30 pts if SimilarWeb returned full traffic data (visits + sources)
  • +25 pts if BuiltWith returned 10+ technologies
  • +20 pts if Apollo returned revenue and headcount
  • +10 pts if at least one ad library returned active ads
  • +10 pts if cloud provider detected via IP match
  • +5 pts per detected ad pixel (capped at +20)
  • HIGH ≥ 70 · MEDIUM 40–69 · LOW < 40
  • Low-confidence ranges suppressed. "Connect for exact" replaces the dollar amount — never show a number we can't stand behind.

9 · Behaviour rules — global

Cross-state rules that apply across the whole conversational flow.

1
Domain alone always works

The agent delivers a complete pitch and signup form on domain alone. Refinement chips are upsell, never blockers.

2
Single-tap inputs only

No free text, no multi-turn questions. Chips and toggles. If a user can't answer in one tap, the question doesn't ship.

3
User input always wins

Self-reported values override inferred values silently. Never argue with the user.

4
Live recalibration

Every chip tap updates the displayed range immediately. No "submit" button. The user sees the number move.

5
Validation language only

Never "estimate," "AI-generated," "inferred." Always "based on patterns across similar accounts."

6
Show the receipt change

When inputs shift the cohort, the source line updates. Smaller N is honest — surface it, don't hide it.

7
Never block on inputs

Signup form is always tappable. If the user signs up without tapping a chip, that's a successful conversion.

8
Big disagreements → soft surface

When user input is wildly inconsistent with inference, agent acknowledges with one line. Never questions the user.

9
Lead with largest opportunity

Multi-ID expansion always surfaces the biggest dollar opportunity first. Users follow the money.

10
Carry context forward

Self-reported values, sizing detection, and chip overrides all promote to the website record on signup. Onboarding never re-asks.

The conversion math

Target metric: ratio of pre-signup estimate to post-connect actual, target band 0.7×–1.3×. If 30% of visitors tap at least one chip and the chips lift estimate accuracy from ±60% to ±25%, the post-signup connect rate should improve materially because the pre-signup number now matches the post-connect actual closely enough to feel honest.
